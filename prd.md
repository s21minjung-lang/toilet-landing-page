# PRD

## 0. 제품 개요

### 0.1 제품 정의
- `everytoilet`은 서울 내 공중/개방/제휴 화장실을 빠르게 탐색하고, 필요한 경우 유료 화장실을 1회성 디지털 티켓으로 이용할 수 있게 하는 위치 기반 서비스다.
- 서비스의 핵심 가치는 `빠른 발견`, `검증된 최신 정보`, `막혔을 때 즉시 대체안 제시`다.
- 무료 화장실 탐색과 유료 화장실 티켓 구매, 방문 인증 기반 정보 갱신을 하나의 흐름으로 연결한다..

### 0.2 해결하려는 문제
- 급한 상황에서 사용자는 검색보다 이동이 우선인데, 현재 화장실 정보는 흩어져 있고 최신성이 낮다.
- 민간 개방 화장실은 실제 이용 가능 여부, 운영 시간, 출입 방식, 유료 여부를 확인하기 어렵다.
- 공공 데이터만으로는 혼잡도, 잠김 여부, 소모품 상태 같은 현장 정보를 반영하기 어렵다.

### 0.3 핵심 타겟 사용자
- 서울 도심에서 이동 중 급하게 화장실을 찾아야 하는 사용자
- 야간이나 상권 밀집 지역에서 즉시 사용 가능한 화장실이 필요한 사용자
- 유아 동반으로 기저귀 교환대 등 부가 시설 정보가 중요한 사용자
- 휠체어 진입 가능 여부 등 접근성 정보가 중요한 사용자

### 0.4 제품 목표
- 첫 진입 후 10초 이내에 근처 화장실 후보를 제시한다.
- 사용자가 3번 이내의 상호작용으로 길안내 또는 티켓 구매까지 도달하게 한다.
- 화장실 상세 정보의 최신성 부족으로 인한 실패 경험을 최소화한다.
- 크라우드소싱과 업주 갱신을 결합해 데이터 최신성을 지속적으로 유지한다.

## 1. 핵심 서비스 UI/UX

### 1.1 디자인 원칙

#### 1.1.1 제품 디자인 목표
- 본 제품은 `전형적인 지도 앱 UI`가 아니라 `프리미엄 금융/라이프스타일 앱 수준의 정제된 모바일 경험`을 목표로 한다.
- 사용자가 처음 보자마자 `급할 때 믿고 쓸 수 있는 서비스`라는 신뢰감을 느끼게 해야 한다.
- 시각 언어는 과도한 장식보다 절제된 타이포그래피, 넉넉한 여백, 높은 대비, 부드러운 모션을 기반으로 한다.
- 모든 화면은 `빠른 판단`, `낮은 피로감`, `한 손 조작`, `즉시 이해`를 우선으로 설계한다.

#### 1.1.2 시각 시스템
- 기본 UI 톤은 깨끗한 화이트/라이트 그레이 기반에 소수의 핵심 강조색만 사용하는 방향으로 설계한다.
- 상태 표현은 단순히 색만 바꾸는 방식이 아니라 텍스트, 아이콘, 배지, 카드 구조를 함께 사용해 명확하게 전달한다.
- 타이포그래피는 화면의 정보 우선순위를 바로 이해할 수 있도록 큰 제목, 명확한 수치, 짧은 보조 문구 중심으로 구성한다.
- 카드, 바텀 시트, 입력창, CTA 버튼은 모두 동일한 라운드 규칙과 그림자 규칙을 따르는 디자인 시스템 위에서 통일한다.
- 불필요한 선, 테두리, 과도한 컬러 태그, 복잡한 아이콘 조합을 피하고 정보 밀도를 낮춘다.

#### 1.1.3 인터랙션 원칙
- 한 화면에는 핵심 CTA를 하나만 강조한다. 나머지 행동은 보조 버튼 또는 텍스트 액션으로 후순위 처리한다.
- 사용자가 자주 하는 행동은 누르기 전에 설명하지 않고, 눌렀을 때 자연스럽게 이해되도록 만든다.
- 바텀 시트, 필터, 체크인, 결제, 신고 같은 주요 액션에는 짧고 부드러운 모션과 햅틱 피드백을 적용한다.
- 로딩 상태는 빈 화면 대신 스켈레톤, 프리뷰 카드, 점진적 콘텐츠 노출로 처리한다.

#### 1.1.4 지도 UI 재정의
- 지도는 `배경 도구`가 아니라 `탐색 캔버스`로 취급한다.
- 기본 지도 SDK에서 제공하는 투박한 줌 버튼, 나침반, 레이어 버튼, 콜아웃 말풍선, 기본 POI 라벨은 그대로 사용하지 않는다.
- 지도 위 정보는 최소화하고, 핵심 해석은 상단 검색/필터와 하단 바텀 시트 카드에서 담당하게 한다.
- 지도의 색과 밀도는 낮추고, 화장실 마커와 현재 위치, 이동 경로만 또렷하게 보이도록 커스텀 스타일을 적용한다.
- 마커를 많이 띄우더라도 화면이 지저분해 보이지 않도록 상태 우선순위 기반의 선택적 노출과 클러스터링을 적용한다.

### 1.2 첫 진입 경험

#### 1.2.1 첫 화면
- 첫 화면은 `현재 위치 중심 지도 홈`으로 시작한다.
- 화면 진입 직후 사용자가 서비스 목적을 이해할 수 있도록 지도, 검색창, 빠른 필터, 근처 화장실 바텀 시트를 즉시 노출한다.
- 첫 화면에서 가장 먼저 보여야 하는 정보는 `지금 열림`, `도보 시간`, `무료/유료`, `최근 검증 시간`이다.
- 사용자가 앱을 처음 실행해도 별도 소개 슬라이드 없이 바로 탐색을 시작할 수 있게 한다.
- 첫 화면에는 배너 캐러셀, 과도한 안내 문구, 광고성 카드, 복수의 동등한 CTA를 두지 않는다.
- 첫 진입 구조는 `검색창`, `필터`, `지도`, `근처 화장실 카드` 네 가지 레이어만 보이도록 단순화한다.

#### 1.2.2 권한 요청 정책
- 위치 권한은 앱 첫 실행 직후 강제 요청하지 않고, 지도 화면이 보인 뒤 맥락과 함께 요청한다.
- 위치 권한 허용 시 현재 위치로 즉시 이동하고, 거부 시에는 서울 중심 지도와 추천 화장실 목록을 기본값으로 제공한다.
- 알림 권한은 필수 권한이 아니며, 환불 결과, 신고 처리, 즐겨찾기 근처 신규 제휴 오픈 알림이 필요할 때 선택적으로 요청한다.

### 1.3 지도 기반 화장실 탐색

#### 1.3.1 한 손 조작 지도 뷰
- 맵 뷰포트를 서울로 제한한다.
- 모든 핵심 액션 버튼을 화면 하단에 고정한다.
- 앱 실행 시 스플래시를 최소화하고 현재 위치를 즉시 로딩한다.
- 바텀 시트를 아래로 스와이프하여 닫을 수 있는 제스처를 지원한다.
- 지도 홈에서 `내 위치`, `새로고침`, `필터 초기화`는 한 손 조작이 가능한 위치에 배치한다.
- 상단 검색창은 고정형 헤더가 아니라 떠 있는 카드형 입력창으로 설계한다.
- 지도 위 컨트롤은 최소한으로 유지하고, 모든 버튼은 시스템 기본형 대신 커스텀 원형 또는 필형 버튼으로 통일한다.

#### 1.3.2 정보 압축형 복합 마커
- 상태별 마커 베이스 컬러 코딩을 적용한다. 예: 개방 중, 혼잡, 유료, 점검 중, `Outdated`
- 마커 오버레이 배지를 적용한다. 예: 휠체어, 기저귀, 자물쇠, 24시간, 최근 검증
- 활성화된 태그 필터의 배지를 가장 크게 노출하는 렌더링 로직을 적용한다.
- SVG 아이콘 동적 조합 렌더링으로 멀티 마커 성능을 최적화한다.
- `Suspend` 상태 시설은 기본 지도에 노출하지 않으며, 운영자 또는 업주 화면에서만 조회 가능하게 한다.
- 마커는 전형적인 핀 모양보다 `카드형 또는 필형 요약 마커`를 우선 검토한다.
- 무료/유료, 혼잡, 검증 상태처럼 즉시 판단에 필요한 정보는 마커 한 번만 보고도 이해할 수 있게 설계한다.
- 확대 레벨에 따라 텍스트 포함 마커, 아이콘 마커, 클러스터 마커로 자연스럽게 전환한다.

#### 1.3.3 상세 정보 및 필터링
- 마커 클릭 시 바텀 시트에 요약 정보를 노출한다. 예: 이름, 상태, 거리, 도보 시간, 무료/유료
- 바텀 시트 확장 시 세부 위치 텍스트, 운영 시간, 주의사항, 이용 조건, 최근 검증 시간을 전체 노출한다.
- 지도 상단에 가로 스크롤 가능한 태그 필터 바(Chip)를 배치한다.
- MVP 기준 기본 필터는 `지금 열림`, `무료`, `유료 가능`, `가까움`, `휠체어`, `기저귀`, `최근 검증`으로 구성한다.
- 검색은 역 이름, 건물명, 동 이름 기준으로 지원한다.
- 필터 칩은 얇은 회색 아웃라인 나열이 아니라 선택 상태가 즉시 구분되는 굵은 세그먼트 스타일을 적용한다.
- 필터 선택 시 지도와 바텀 시트가 동시에 부드럽게 갱신되어야 하며, 화면 전체 리렌더로 보이면 안 된다.

#### 1.3.4 상태별 화면 처리
- 위치 로딩 중에는 지도 스켈레톤과 바텀 시트 플레이스홀더를 노출한다.
- 반경 내 결과가 없으면 `근처에는 없어요` 메시지와 함께 검색 반경 확장 또는 인기 지역 이동 CTA를 제공한다.
- `Outdated` 상태는 회색 마커와 경고 문구로 명확히 구분하고, 티켓 구매 CTA를 비활성화한다.
- 서비스 외 지역에서는 서울 진입 전까지 제보와 체크인 기능을 비활성화한다.
- 빈 상태, 오류 상태, 권한 거부 상태도 모두 동일한 디자인 언어의 카드형 안내 UI로 처리한다.

### 1.4 화장실 상세 화면

#### 1.4.1 정보 우선순위
- 상세 화면 최상단에는 이름, 상태, 도보 시간, 거리, 최근 검증 시간을 요약 노출한다.
- 그 아래에는 주소, 운영 시간, 이용료, 접근성, 소모품/청결/혼잡 관련 최근 상태를 순서대로 배치한다.
- 무료 화장실은 `길안내`와 `체크인` CTA를 우선 노출하고, 유료 화장실은 `티켓 구매` CTA를 우선 노출한다.
- 상세 화면은 `지도 위 작은 말풍선`이 아니라 별도 확장형 바텀 시트 또는 풀스크린 디테일 카드 경험으로 설계한다.
- 상세 카드 상단에는 대표 정보만 남기고, 나머지 세부 정보는 섹션 카드로 구분해 읽기 쉽게 정리한다.

#### 1.4.2 신뢰 배지
- 각 화장실 상세 카드에는 정보 출처 배지를 노출한다. 예: `공공데이터`, `업주 인증`, `유저 검증`
- 정보 항목마다 마지막 검증 시점과 갱신 주체를 기록한다.
- 서로 다른 출처의 정보가 충돌할 경우 해당 항목에 `검증 중` 상태를 노출한다.

#### 1.4.3 티켓 및 결제 화면 디자인
- 결제 화면은 복잡한 폼 나열 대신 `시설 정보`, `가격`, `유효 시간`, `결제 수단`, `핵심 CTA` 순서의 단일 흐름으로 설계한다.
- 티켓 화면은 시각적으로 가장 신뢰감 있는 화면이어야 하며, 비밀번호, 만료 시간, 문제 신고 버튼이 한눈에 구분되어야 한다.
- 티켓 화면의 비밀번호 영역은 대비를 높이고 여백을 넉넉하게 사용해 읽기 오류를 줄인다.
- 긴급 신고 버튼은 강하게 보여야 하지만 전체 화면의 미감을 깨지 않도록 구조적으로 분리된 위험 액션 영역으로 배치한다.

## 2. 검증 및 비즈니스

### 2.1 화장실 티켓 중개 시스템

#### 2.1.1 업주 직접 등록 및 관리
- 업주용 대시보드를 제공한다. 상세 정보와 주기적인 비밀번호 등록 폼을 포함한다.
- 업주 등록 시 필수 입력값은 시설명, 주소, 위치 좌표, 운영 시간, 이용 요금, 편의시설, 출입 방식, 고객 문의 연락처다.
- 업주가 등록한 시설은 운영자 승인 후 판매 가능 상태로 전환한다.
- 도어락 비밀번호는 즉시 변경 또는 예약 변경 방식으로 관리할 수 있게 한다.
- 비밀번호 변경 이력과 변경 시각은 운영자와 업주가 확인할 수 있도록 로그로 저장한다.
- 1회 이용 요금 설정 기능을 제공한다. 예: 1,000원
- 결제액의 80%는 업주 정산, 20%는 플랫폼 수수료로 처리한다.

#### 2.1.2 간편 결제 및 티켓 발급
- 토스 미니앱 환경에 맞춘 토스페이(Toss Pay) 원클릭 간편결제를 연동한다.
- 결제 완료 시 1회용 디지털 티켓 팝업을 노출한다.
- 티켓 화면 중앙에 현재 유효한 도어락 비밀번호를 크게 노출한다.
- 티켓에는 시설명, 결제 시각, 유효 만료 시각, 환불/신고 버튼, 이용 주의사항을 함께 표기한다.
- 1장의 티켓은 1인 1회 입장을 기본 정책으로 한다.
- 티켓 유효시간은 구매 시점 기준 30분으로 설정하며, 만료 이후에는 재노출되지 않는다.
- 동일 사용자의 동일 시설 중복 구매는 최근 구매 티켓이 유효한 동안 제한한다.

#### 2.1.3 환불 및 예외 정책
- `비밀번호 틀림`, `문 잠김`, `시설 접근 불가` 신고는 티켓 유효시간 내에만 접수 가능하게 한다.
- 긴급 신고가 접수되면 우선 환불을 즉시 실행하고, 운영자는 사후 검토를 진행한다.
- 비밀번호를 확인하기 전 사용자가 결제를 취소한 경우에는 즉시 전액 환불한다.
- 비밀번호가 노출된 이후에는 원칙적으로 환불 불가이며, 장애 또는 허위 정보가 확인된 경우 예외적으로 환불한다.
- 환불 사유, 환불 시각, 대상 시설, 업주 차감 여부는 모두 정산 로그에 남긴다.

#### 2.1.4 정산 정책
- 업주 정산은 주 단위 정산을 기본으로 한다.
- 환불이 발생한 거래는 해당 주 정산 금액에서 우선 차감한다.
- 반복 신고로 인해 `Suspend` 처리된 시설은 운영자가 해제할 때까지 신규 판매와 정산 확정을 중단한다.

### 2.2 데이터 최신화 및 검증

#### 2.2.1 업데이트 갱신 강제
- UI 최상단에 마지막 업데이트 시간(LastVerified) 배지를 노출한다.
- 시설 화장실 정보가 7일 이상 미갱신되면 `Outdated` 상태(회색 마커)로 자동 변경한다.
- `Outdated` 상태에서는 결제 및 티켓 구매 버튼을 시스템적으로 비활성화한다.
- 30일 이상 미갱신된 제휴 화장실은 일반 사용자 검색 결과에서 우선순위를 크게 낮춘다.

#### 2.2.2 공공 데이터 동기화
- 서울시 공중/개방 화장실 API를 주 1회 정기 호출하고 폐쇄/변경 상태를 DB에 반영한다.
- 공공 데이터 동기화 시 신규 등록, 폐쇄, 운영 시간 변경, 좌표 보정 등 변경 유형을 분리해 기록한다.
- 동기화 실패 시 운영자에게 즉시 알림을 보내고 마지막 성공 시각을 운영 콘솔에 표기한다.

#### 2.2.3 크라우드소싱 검증 반영 규칙
- 유저 설문 결과는 동일한 항목에 대해 최근 7일 내 3명 이상의 독립 사용자 응답이 일치할 때 자동 반영한다.
- 상충 응답이 반복되면 해당 항목은 자동 반영하지 않고 `검증 중` 상태로 유지한다.
- 이용료, 도어락 비밀번호, 운영 시간 같은 민감 정보는 업주 또는 운영자 검증 없이는 유저 응답만으로 즉시 반영하지 않는다.
- 소모품 여부, 청결 상태, 혼잡도는 유저 응답 기반으로 우선 반영 가능하게 한다.

#### 2.2.4 위치 인증 및 부정행위 방지
- 유저 GPS와 화장실 마커 간 거리를 계산해 반경 30m 이내일 때만 체크인을 허용한다.
- GPS 정확도가 지나치게 낮거나 비정상 이동 패턴이 감지되면 체크인을 차단한다.
- 동일 계정의 반복 체크인, 동일 디바이스 다중 계정 응답, 비정상 속도 이동은 리스크 이벤트로 기록한다.
- 유저 위치가 서울 외 지역일 경우 체크인 및 제보 기능을 전체 비활성화한다.

#### 2.2.5 긴급 신고 및 판매 중단 제어
- 디지털 티켓 화면 내 빨간색 `긴급 신고(비밀번호 틀림/문 잠김)` 버튼을 배치한다.
- 최근 24시간 내 동일 시설에서 2건 이상의 긴급 신고가 누적되면 자동 `Suspend` 처리한다.
- 신고 누적 시 해당 화장실 판매를 즉시 중단(Suspend)하고 갱신 요청 알림을 보낸다.
- 신고 접수 시 피해 유저에게 즉시 환불 처리하는 로직을 연동한다.
- `Suspend` 해제는 업주 정보 갱신 후 운영자 검토를 거쳐 수동 승인한다.

### 2.3 운영자 관리 기능

#### 2.3.1 운영 콘솔
- 운영자는 시설 상태, 최근 신고, 환불 현황, 동기화 오류, `Outdated` 비율을 한눈에 볼 수 있어야 한다.
- 운영 콘솔에서 시설 강제 숨김, 강제 노출, 수동 환불, 신고 상태 변경, 업주 연락 이력 기록이 가능해야 한다.

#### 2.3.2 감사 로그
- 비밀번호 변경, 티켓 발급, 신고 접수, 자동 반영, 수동 수정, 환불 실행은 모두 로그로 남긴다.
- 로그는 운영 이슈 추적과 분쟁 대응을 위해 최소 6개월 이상 보관한다.

## 3. 유저 참여(크라우드소싱)

### 3.1 체크인 및 리뷰 시스템

#### 3.1.1 위치 기반 방문 인증
- 유저 GPS와 화장실 마커 간 거리를 계산해 반경 30m 이내일 때만 체크인을 허용한다.
- 체크인 완료 후 설문 또는 상태 제보를 이어서 진행할 수 있게 한다.
- 체크인 이후 일정 시간 내에만 설문 응답을 유효 응답으로 인정한다.

#### 3.1.2 동적 맞춤형 설문
- DB 스캔 후 결측치(`null`) 또는 가장 노후화된 정보의 우선순위를 계산하는 알고리즘을 적용한다.
- 계산된 우선순위에 따라 3개의 설문 질문만 팝업으로 노출한다. 예: `휴지 있나요?`
- 질문은 10초 이내 응답 가능한 객관식 위주로 구성한다.
- 다수 유저의 동일 응답이 누적되면 DB 상태에 자동 반영한다.

#### 3.1.3 포인트 보상 및 전환
- 설문 완료 시 즉각적인 시각 효과와 함께 인앱 포인트를 소액 지급한다.
- 정보 가치가 높은 응답일수록 추가 포인트를 지급할 수 있다. 예: 신규 시설 첫 검증, 오래된 정보 갱신
- 적립 포인트를 토스 포인트로 전환하거나 결제 시 할인 포인트로 사용할 수 있게 한다.
- 부정 적립 방지를 위해 일일 적립 한도를 둔다.

#### 3.1.4 구조화된 제보
- 자유 서술형 리뷰보다 구조화된 상태 제보를 우선 지원한다. 예: `휴지 없음`, `청결 불량`, `문 잠김`, `혼잡`
- 운영 부담을 줄이기 위해 MVP에서는 사진 업로드와 긴 자유 텍스트 리뷰는 제외한다.

## 4. 핵심 사용자 플로우

### 4.1 무료 화장실 탐색 플로우
- 사용자는 앱 실행 후 현재 위치 중심 지도 홈을 본다.
- 필터 또는 검색으로 원하는 조건의 화장실을 찾는다.
- 마커 탭 후 바텀 시트에서 상세 정보를 확인한다.
- `길안내`를 눌러 이동한다.
- 방문 후 체크인과 설문으로 최신 정보를 보완한다.

### 4.2 유료 화장실 티켓 구매 플로우
- 사용자는 유료 화장실 상세 화면에서 가격과 최근 검증 정보를 확인한다.
- `티켓 구매`를 눌러 토스페이 결제를 진행한다.
- 결제 완료 후 즉시 디지털 티켓과 도어락 비밀번호를 확인한다.
- 문제가 발생하면 티켓 화면에서 긴급 신고 및 환불을 진행한다.

### 4.3 데이터 검증 플로우
- 사용자는 현장 도착 후 체크인한다.
- 시스템은 해당 시설의 결측치와 노후 정보를 계산해 질문 3개를 제시한다.
- 응답은 포인트 지급과 함께 저장되고, 조건 충족 시 DB에 자동 반영된다.

### 4.4 장애 대응 플로우
- 사용자가 티켓 사용 중 비밀번호 오류 또는 문 잠김을 겪는다.
- 티켓 화면에서 긴급 신고를 누른다.
- 시스템은 즉시 환불을 실행하고 해당 시설 판매를 일시 중단할지 판단한다.
- 운영자는 로그와 누적 신고를 검토해 시설 상태를 수정하거나 판매를 재개한다.

## 5. 정책 및 비기능 요구사항

### 5.1 개인정보 및 보안
- 위치 정보는 서비스 제공과 방문 인증 목적에 한해 최소 범위로 수집한다.
- 도어락 비밀번호는 저장 시 암호화하고, 운영 로그에는 평문 전체를 남기지 않는다.
- 결제 및 환불 관련 데이터는 분쟁 대응을 위해 안전하게 보관한다..

### 5.2 성능 요구사항
- 첫 화면 지도와 근처 목록은 앱 진입 후 2초 내에 기본 상태를 노출해야 한다.
- 마커가 많은 지역에서도 지도 스크롤과 줌이 끊기지 않도록 렌더링 최적화를 적용한다.
- 바텀 시트 전환과 마커 선택 반응은 즉각적이어야 한다.

### 5.3 접근성 요구사항
- 컬러 외에 텍스트와 아이콘으로 상태를 함께 전달한다.
- 주요 CTA는 한 손 조작이 가능한 크기와 위치를 유지한다.
- 스크린리더 사용자를 위해 마커, 필터, CTA에 명확한 접근성 라벨을 제공한다.

### 5.4 디자인 시스템 및 품질 기준
- 제품 전반에 공통으로 적용되는 디자인 토큰을 정의한다. 예: 색상, 타이포그래피, 간격, 라운드, 그림자, 모션 시간, 햅틱 규칙
- 운영체제 기본 지도 UI 컴포넌트와 기본 다이얼로그를 가능한 한 그대로 노출하지 않고, 제품 디자인 시스템 안에서 재정의한다.
- 모든 신규 화면은 `정보 위계`, `CTA 우선순위`, `여백`, `모션`, `접근성`, `로딩 상태` 검수를 통과해야 릴리즈할 수 있다.
- 각 화면은 `한눈에 이해되는가`, `한 손으로 조작 가능한가`, `기본 지도 앱보다 더 깔끔한가`를 기준으로 디자인 리뷰를 진행한다.
- 사용자에게 직접 보이는 핵심 화면인 지도 홈, 상세 화면, 결제 화면, 티켓 화면은 별도 고해상도 QA 대상 화면으로 관리한다.
